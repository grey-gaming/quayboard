# Current Milestone

## Status

M5: Feature Workstream Specifications

This milestone is the active implementation target.

## Goal

Deliver the first full Feature Editor on top of the completed M4 catalogue: five independent feature workstream tracks, version history, approvals, generation from approved project specs, and Mission Control updates for feature-document readiness.

## In Scope

- `feature_product_specs`, `feature_product_revisions`, `feature_ux_specs`, `feature_ux_revisions`, `feature_tech_specs`, `feature_tech_revisions`, `feature_user_doc_specs`, `feature_user_doc_revisions`, `feature_arch_doc_specs`, and `feature_arch_doc_revisions` schema additions
- feature workstream revision history, manual authoring, and independent approval for Product, UX, Tech, User Docs, and Architecture Docs
- product-level workstream requirement flags (`ux_required`, `tech_required`, `user_docs_required`, `arch_docs_required`) and tab visibility driven from the current head Product revision
- `GET /features/:id/tracks` plus workstream revision, generation, and approval routes
- `GenerateFeatureProductSpec`, `GenerateFeatureUxSpec`, `GenerateFeatureTechSpec`, `GenerateFeatureUserDocs`, and `GenerateFeatureArchDocs` job support with explicit validation before persistence
- frontend route for `/projects/:id/features/:featureId`, including five workstream tabs plus the M5 Tasks stub tab
- Mission Control phase-gate and next-action updates for feature product-spec readiness
- user-facing and architecture docs that describe the M5 repo reality

## Out Of Scope

- standalone task planning routes or delivery-task generation
- feature review-item triage, refinement loops, or real ReviewPanel findings
- bug report workflows, implementation record writes, or live implementation-staleness tracking
- sandbox execution, PR creation, or verification runs
- generic graph editing beyond direct `depends_on` dependencies
- OAuth, RBAC, API keys, or other M12 auth extensions

## Acceptance Criteria

The milestone is complete when the repo can support the following:

- A feature has independent Product, UX, Tech, User Docs, and Architecture Docs workstreams with navigable revision history
- Each workstream can be authored manually or generated by LLM
- Product revisions persist workstream requirement flags and hide/show downstream tabs correctly
- `GET /features/:id/tracks` returns the current head and approval state of all five workstreams
- Feature workstream generation uses the approved project Product Spec, UX Spec, and Technical Spec rather than the one-pager
- Feature approvals are independent and only the current head revision counts as approved for track status
- Mission Control’s Features phase requires at least one feature with an approved Product workstream revision
- The Feature Editor page stays behind the existing approved-user-flows gate and includes the M5 Tasks stub without adding M6 task routes
- `pnpm typecheck`, `pnpm test`, and `pnpm build` pass
- Architecture and user docs describe the M5 repo behavior

## Working Rules

- Treat the approved project Product Spec, UX Spec, and Technical Spec as the source set for feature workstream generation.
- Keep feature workstream approvals independent; do not collapse them back into a single feature-level approval switch.
- Keep the Tasks tab present as a stub only; do not add the M6 standalone task routes or delivery-task data model in production code.
- Keep live implementation-staleness tracking deferred until M6 implementation-record support exists.
- If implementation needs to deviate further from the outline’s cookie-session, SSE, or project-scoped secret model, capture that in an ADR first.
